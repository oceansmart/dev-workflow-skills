# /jira:show - 이슈 상세 조회 & 요구사항 명확화

## Usage
```
/jira:show [TICKET-ID]
```

## Parameters
- **TICKET-ID**: Jira 이슈 키 (예: SMFD-184)

## 목적 및 역할

**이 명령은 두 가지 시나리오에서 사용됩니다:**

1. **단순 조회**: 티켓 내용이 명확할 때 → 정보 확인 후 바로 `/jira:start` 진행
2. **요구사항 명확화**: 티켓 내용만으로 작업 파악이 어려울 때 → Socratic 대화로 명확한 작업계획 수립

```
워크플로우 위치:
/jira:list → /jira:show (요구사항 명확화) → /jira:start (명확한 계획으로 시작)
                 ↓
         [Socratic 대화]
                 ↓
         [작업계획 확정]
```

## Phase 0: 설정 파일 확인 (필수 - BLOCKING)

**⛔ 이 단계를 통과하지 않으면 어떤 작업도 진행할 수 없습니다.**

1. `.claude/config/jira.md` 파일 존재 여부 확인
2. **파일 있음**: 설정값 로드 → Phase 1 진행
3. **파일 없음**: 아래 메시지 출력 후 **완전 중단**

```
⛔ Jira 설정 파일이 없습니다.

/jira:* 명령어를 사용하려면 먼저 설정 파일이 필요합니다.

📁 필요한 파일: .claude/config/jira.md

설정 파일을 생성할까요? (예/아니오)

→ "예" 선택 시: 설정 정보 입력 → 파일 생성 → 다시 명령어 실행
→ "아니오" 선택 시: 명령어 종료
```

**⚠️ 설정 파일이 생성되기 전까지 Phase 1 이후 단계는 절대 실행 불가**

## Phase 1: 조회 확인 요청

```
📋 티켓 조회 확인

티켓: [TICKET-ID]

수행 작업:
1. 티켓 상세 정보 조회
2. 첨부 이미지 분석 (있는 경우)
3. 요구사항 명확성 판단
4. 불명확 시 Socratic 대화로 명확화

위 작업을 진행할까요? (예/아니오)
```
**→ 사용자 승인 후에만 Phase 1 진행**

## Phase 1: 티켓 분석

### 1.1 이슈 정보 수집
- `jira_get_issue` 호출하여 상세 정보 가져오기
- 모든 필드 포함 (설명, 댓글, 첨부파일 등)

### 1.2 텍스트 + 이미지 동등 분석

티켓 이해를 위해 **텍스트 설명과 첨부 이미지는 동등하게 중요한 단서**입니다.
어느 하나도 지나치지 말고 **면밀하게 분석하고 판단**해야 합니다.

**텍스트 분석:**
- 제목, 설명, 댓글 내용 파악
- 요구사항, 문제 상황, 기대 결과 정리

**이미지 분석 (필수):**
- 첨부된 이미지가 있으면 `jira_download_attachments`로 다운로드
- `Read` 도구로 각 이미지를 열어 **면밀히 분석**
- 이미지에서 얻을 수 있는 정보:
  - 현재 UI 상태와 문제 발생 위치
  - 에러 메시지, 데이터 예시, 흐름도
  - 텍스트만으로는 파악 불가능한 시각적 맥락

⚠️ **이미지를 건너뛰면 전체 맥락 파악 불가 - 반드시 확인!**

### 1.3 연관 정보 조회
- 에픽 링크 확인
- 하위 이슈 확인
- 연결된 이슈 확인

### 1.4 명확성 판단

**분석 후 판단 기준:**

| 상태 | 조건 | 다음 단계 |
|------|------|----------|
| ✅ **명확함** | 요구사항, 개발범위, 예상파일 모두 파악 가능 | Phase 4로 이동 (바로 start 연결) |
| ⚠️ **불명확함** | 질문이 필요한 부분 존재 | Phase 2 Socratic 대화 시작 |

**불명확 징후 예시:**
- "기능 개선" 같은 모호한 표현
- 구체적인 UI/API 명세 없음
- 영향 범위 파악 불가
- 기존 코드 변경인지 신규 구현인지 불분명

## Phase 2: Socratic 대화 (요구사항 명확화)

**목적**: 반복적인 질문을 통해 불명확한 요구사항을 구체화

### 2.1 질문 카테고리

#### 🎯 목적/범위 질문
```
- 이 기능의 최종 목표가 무엇인가요?
- 어떤 문제를 해결하려고 하나요?
- 영향 범위는 어디까지인가요? (특정 화면? 전체 시스템?)
```

#### 🖥️ UI/UX 질문
```
- 이 기능은 어떤 화면에서 사용되나요?
- 버튼/메뉴의 위치는 어디인가요?
- 사용자 인터랙션 흐름은 어떻게 되나요?
- 성공/실패 시 어떤 피드백을 보여주나요?
```

#### 🔌 API/데이터 질문
```
- 필요한 API 엔드포인트는 무엇인가요?
- 요청/응답 데이터 형식은 어떻게 되나요?
- 기존 API를 수정하나요, 새로 만드나요?
- 연관된 테이블/엔티티는 무엇인가요?
```

#### 🔗 의존성 질문
```
- 기존 유사 기능이 있나요? (참고할 코드)
- 이 작업이 다른 기능에 영향을 주나요?
- 선행 작업이 필요한가요?
```

#### ⚙️ 기술적 질문
```
- 특별한 기술 요구사항이 있나요?
- 성능 요구사항이 있나요?
- 에러 처리 방식은 어떻게 하나요?
```

### 2.2 대화 진행 방식

```
🔄 반복 프로세스:

1. 불명확한 부분 식별 → 질문 제시
   ↓
2. 사용자 답변 수신
   ↓
3. 답변 분석 → 추가 질문 필요 여부 판단
   ↓
4. 추가 질문 있으면 → 1번으로
   모든 것이 명확하면 → Phase 3으로
```

### 2.3 대화 예시

```
📋 SMFD-286: Location API 추가

❓ 질문 1: 이 API는 어떤 화면에서 사용되나요?
   → 사용자: "Master Data 메뉴의 Location 관리 화면입니다"

❓ 질문 2: CRUD 중 어떤 기능이 필요한가요?
   → 사용자: "조회(리스트, 상세)와 수정만 필요합니다"

❓ 질문 3: 기존 Location 관련 코드가 있나요?
   → 사용자: "VesselController 참고하면 될 것 같아요"

❓ 질문 4: 페이지네이션이 필요한가요?
   → 사용자: "네, 기본 20개씩"

✅ 모든 요구사항이 명확해졌습니다. 작업계획을 정리하겠습니다.
```

## Phase 3: 작업계획 정리

### 3.1 확정된 요구사항 정리

```
📋 확정된 요구사항

## 기능 요약
- [명확하게 정리된 기능 설명]

## 상세 요구사항
1. [요구사항 1]
2. [요구사항 2]
3. [요구사항 3]

## 제외 범위 (해당 시)
- [이번 작업에서 제외되는 것]
```

### 3.2 개발 타입 결정

| 타입 | 설명 | 주요 작업 |
|------|------|----------|
| `backend-api` | API 개발 | Controller, Service, Repository, DTO |
| `frontend` | 화면 개발 | Page, Store, API 모듈, 컴포넌트 |
| `database` | DB 작업 | DDL, DML, 마이그레이션 |
| `bugfix` | 버그 수정 | 원인 분석 → 최소 수정 |

### 3.3 예상 작업 파일

```

Frontend:
- [ ] LocationPage.tsx (신규)
```

## Error Handling
- 이슈가 존재하지 않으면 에러 메시지 표시
- 권한이 없으면 접근 권한 안내
- API 오류 시 상세 에러 메시지 표시

## ⛔ 종료 규칙 (CRITICAL)

**/jira:show는 작업계획 확정 후 반드시 종료됩니다.**

- ❌ 자동으로 `/jira:start` 실행 **금지**
- ❌ 자동으로 코드 수정 시작 **금지**
- ⚠️ "👉 다음 단계"는 **안내일 뿐**, 자동 실행 대상 아님
- ✅ 사용자가 **`/jira:start` 명령을 직접 입력**해야만 다음 단계 진행
